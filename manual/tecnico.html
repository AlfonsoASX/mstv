Arranque

Instalación local

Requisitos

  • PHP 8.x o compatible con `mysqli`.
  • MySQL o MariaDB.
  • Extensión PHP `mysqli`.
  • Extensión GD o funciones de imagen para validación facial básica.
  • Servidor local o hosting que permita PHP procedural.

URLs principales

/control/index.php Panel administrativo

/app/index.html App móvil web

/manual/index.html Manuales estáticos

Levantar en puerto 7878

cd /Users/usuario/Documents/GitHub/mstv/control
php -S localhost:7878 -t .

Configuración de base de datos

El proyecto actual usa archivos `db.php` en `control/lib/` y `app/`. Para documentar o instalar en otro ambiente, usa placeholders y evita publicar secretos reales. 

<?php
$db_host = "TU_HOST_MYSQL";
$db_user = "TU_USUARIO_MYSQL";
$db_pass = "TU_PASSWORD_MYSQL";
$db_name = "TU_BASE_DE_DATOS";

$conexion = mysqli_connect($db_host, $db_user, $db_pass, $db_name);
if (!$conexion) {
    die("Error de conexión a la base de datos: " . mysqli_connect_error());
}
mysqli_set_charset($conexion, "utf8mb4");
No publicar credenciales: si existen credenciales reales en código fuente, deben migrarse a variables de entorno o archivos privados fuera del repositorio antes de publicar.

Arquitectura

Estructura general del proyecto

MSTV Control no usa un framework MVC moderno; es una aplicación PHP procedural con dos frontends, librerías compartidas y assets de la plantilla Cork. 

/
├── app/                  # App móvil web y API JSON
├── control/              # Panel administrativo PHP
│   ├── lib/              # Bootstrap, helpers, payroll, schema y operaciones
│   ├── partes/           # Nav, menú y footer
│   └── uploads/          # Fotos base y adjuntos
├── manual/               # Wiki estática de usuario y técnica
├── src/                  # Bootstrap, plugins y assets usados por Cork
├── layouts/              # Layouts compilados de Cork
├── scss/                 # Fuentes SCSS del tema
└── cork-v4.0.0/          # Plantilla original y documentación

Presentación

`control/*.php` y `app/index.html` renderizan pantallas.

Dominio

`control/lib/payroll.php` concentra nómina, vacaciones, caja, préstamos y cálculos.

Datos

MySQL/MariaDB vía `mysqli`, con migraciones ligeras en runtime.

La pantalla consulta agregados por periodo y rol visible. Al modificarla, evita mezclar reglas de negocio con presentación: mantén filtros explícitos, alias claros y protección de visibilidad por cliente/sitio cuando aplique.

`turnos-asignacion.php` debe crear filas reales porque `app/api.php` busca turnos materializados. Cualquier cambio al generador debe preservar sitio obligatorio, rango de fechas y prevención de duplicados por persona/hora inicio.

  • No cambiar contrato móvil si solo cambia la UI de Control.
  • Confirmar que calendario y nómina lean las mismas filas.

Flujo técnico

La acción `checar` valida sesión móvil, turno vigente, ventana de anticipación, geocerca y evidencia. Si no puede calcular reconocimiento facial, debe conservar evidencia y marcar revisión humana en lugar de fallar silenciosamente.

  • Guardar siempre hora real y evento recibido.
  • Actualizar `turnos.hora_entrada_real` o `hora_salida_real` cuando corresponda.
  • Registrar bitácora ante aceptación, rechazo o pendiente.

  • Evitar payloads enormes o binarios.
  • Conservar compatibilidad con vistas agregadas.

Datos en movimiento

Flujos de datos principales

Login administrativo: `control/index.php` valida usuario y rol, crea sesión PHP y redirige al dashboard.
Render de página moderna: página PHP carga `lib/app.php`, valida sesión/permisos, consulta MySQL y renderiza con `lib/ui.php`.
Checada móvil: app obtiene turno, captura GPS/selfie, API valida ventana, geocerca y evidencia, inserta `registros_asistencia` y actualiza `turnos`.
Incidencia: app crea registro, Control lo lista y supervisión actualiza estado/seguimiento.
Nómina: cálculo toma turnos, checadas, faltas, descansos, vacaciones, caja, préstamos y capturas para persistir resumen/conceptos.
Permisos: roles guardan matriz JSON; menú y acceso directo validan sección/página.
Operación segura

Seguridad y mantenimiento

Credenciales: no deben publicarse en documentación ni repositorios. Usar placeholders en manuales y migrar a variables de entorno cuando sea posible.
Backups: respaldar base de datos y carpetas `app/uploads` y `control/uploads` antes de cambios grandes.
Runtime schema: para producción, validar `APP_SCHEMA_BOOTSTRAP` y probar cambios en copia de base.
Permisos: el rol DUEÑO conserva acceso total; los demás dependen de matriz por rol.
Extensión del sistema

Guía para programadores

Agregar una página nueva a Control
  1. Crear archivo PHP en `control/`.
  2. Incluir `require_once __DIR__ . '/lib/app.php';` y, si usa layout nuevo, `lib/ui.php`.
  3. Aplicar `app_require_session()` y `app_require_page_permission()` o roles específicos.
  4. Agregar la página al catálogo en `app_permission_catalog()` si debe controlarse por permisos.
  5. Agregar entrada de menú solo si será visible.
Agregar una configuración global
  1. Agregar clave default en `runtime_schema.php` dentro de `$defaults`.
  2. Leerla con `app_config_int`, `app_config_float` o `app_config`.
  3. Usarla en lógica de negocio sin hardcodear valores.
  4. Documentarla en este manual si afecta operación.
Modificar nómina

Centraliza cambios en `control/lib/payroll.php` y evita duplicar cálculos en pantallas. Persiste conceptos en `nomina_conceptos` para que recibos/exportación expliquen el resultado.

Actualizar esta wiki

Edita los HTML de `manual/` y usa los componentes existentes: `section`, `card`, `steps`, `accordion`, `table-wrap`, `code-block`, `callout` y `tabs`.
